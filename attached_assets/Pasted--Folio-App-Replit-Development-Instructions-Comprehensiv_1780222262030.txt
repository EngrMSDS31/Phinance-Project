#### Transaction Type Logic Per Asset Type

Each asset TYPE must restrict which transaction types are available when the user adds a transaction to that holding. Apply this logic **globally** — in every place transactions are added (the "Add Transaction" modal, inline transaction forms, etc.).

| Asset Type | Allowed Transaction Types |
|---|---|
| **Savings Account** | Deposit, Withdrawal, Dividend/Interest, Other Income, Other Expenses |
| **Cash** | Deposit, Withdrawal, Other Income, Other Expenses |
| **Stocks** | Buy, Sell, Dividend, Stock Split, Other Income, Other Expenses |
| **Bonds** | Buy, Sell, Coupon/Interest, Maturity, Other Income, Other Expenses |
| **Cryptocurrencies** | Buy, Sell, Staking Reward, Transfer, Other Income, Other Expenses |
| **ETFs/Funds** | Buy, Sell, Distribution/Dividend, Other Income, Other Expenses |
| **Other Custom Assets** | Deposit, Withdrawal, Other Income, Other Expenses |

**Key rule:** For asset types that have **no Buy/Sell** (Savings Account, Cash, Other Custom Assets):
- There are **no Capital Gains** and **no Realized Gains** calculations for those holdings.
- They still count as **assets within the portfolio** — they contribute to Total Portfolio Value and Current Value calculations.
- They are **NOT treated as idle cash** unless explicitly tagged as "Idle Cash" by the user.
- Their returns are tracked via: Cash Incentive = Sum of (Income transactions) − Sum of (Expense transactions).

**Implementation note for Replit:** Store the `assetType` on each holding record. When the "Add Transaction" modal/form opens for a holding, read its `assetType` and dynamically filter the transaction type dropdown to only show allowed types for that asset type. This filter must apply both on the frontend (UI) and backend (validation).

### 1.2 Base Currency Default for All Portfolios

**Rule:** When a portfolio is created, the user sets a **Base Currency**. This base currency must become the default selected currency for every new holding added to that portfolio and every transaction within it.

**Changes required:**

1. On the "Create Portfolio" screen, ensure the Base Currency selector is prominent and required.
2. When the "Add Holding" modal opens within a portfolio, pre-select the portfolio's base currency in the CURRENCY dropdown.
3. The user can still override the currency per holding (e.g., a USD-denominated ETF inside a PHP-base portfolio), but the default must always be the portfolio base currency.
4. Apply this rule to **all portfolio types** — not just custom portfolios. Mixed-type portfolios, stock portfolios, and all others must follow this same default behavior.

## 2. DASHBOARD SECTION — UI & CALCULATION CHANGES

### 2.1 Metric Card Logic Changes

The four metric cards on the Dashboard need their **labels and calculation logic** revised as follows.

#### Card 1: "CURRENT VALUE" → Rename to **"TOTAL PORTFOLIO"**
```
Total Portfolio = Current Value (assets only) + Total Cash (idle cash)
```
- Shows the grand total across all portfolios.
- Sub-label: Keep "Invested: ₱X" and percentage return as-is.

#### Card 2: "TOTAL PROFIT" → Keep label as **"TOTAL PROFIT"**
```
Total Profit = Capital Gain + Realized Gains + Dividends + Cash Incentive

Where:
  Capital Gain     = Unrealized gain on open positions
  Realized Gains   = Gain from sold positions
  Dividends        = All dividend/income transactions in Holdings > Returns
  Cash Incentive   = (Total Cash Income transactions) − (Total Cash Expense transactions)
```
- Sub-label: Keep percentage display.

#### Card 3: "TOTAL CASH" → Rename to **"CURRENT VALUE"**
```
Current Value = Total Assets Only (sum of all holdings at current price)
             — Idle Cash is EXCLUDED from this figure
```
- Sub-label: Keep "+X% of total" or show as "+₱X" of portfolio.

#### Card 4: "DIVIDENDS" → Rename to **"TOTAL CASH"**
```
Total Cash = Idle Cash only
           (Cash balances not invested in any asset)
```
- Sub-label: Keep "+X% of total" display.

**Summary of card order and new labels:**

| Position | Old Label | New Label | New Calculation |
|---|---|---|---|
| Top-left | Current Value | Total Portfolio | Assets + Idle Cash |
| Top-right | Total Profit | Total Profit | Cap Gain + Realized + Dividends + Cash Incentive |
| Bottom-left | Total Cash | Current Value | Assets only, no idle cash |
| Bottom-right | Dividends | Total Cash | Idle cash only |

### 3.3 Dividends Chart — Fix Data Accuracy

**Current issue:** The dividend bar chart (Monthly / Yearly / Asset toggle) is displaying inaccurate values.

**Required fix:**

Each bar in the chart must reflect the **Net Dividend Value** for that period or asset. Net Dividend Value = sum of all dividend/distribution/interest transactions received (after any applicable withholding tax if tracked).

Rules per toggle:
- **Monthly view:** Each bar = total net dividends received in that calendar month for the selected year. The year selector (← 2026 →) controls which year is shown. All 12 months must always render (bars with zero value show as empty/flat — not hidden).
- **Yearly view:** Each bar = total net dividends for that calendar year. Show at minimum the last 5 years of data, or all years with data — whichever is greater.
- **Asset view:** Each bar = total net dividends received from that specific holding/asset, across all time (or the selected year if a year filter is active).

**Verification checklist for Replit:**
1. Cross-check the total shown in each bar against the raw dividend transaction records.
2. Ensure currency conversion is applied consistently if holdings are in different currencies (convert to base currency before summing).
3. Ensure dividends from non-tradable holdings (Savings Account interest, etc.) are included if they are tagged as "Dividend/Interest" transactions.

**Replacement behavior:** All data must refresh automatically when:

### 5.3 PSE (Philippine Stock Exchange) Price Fetching — Optimize EODHD API Calls

**Current issue:** The app is making excessive EODHD API calls and has an artificial 20-call-per-day limit in the code.

**Required changes:**

#### A. Only fetch prices for active holdings
```
Active holding = any stock in the user's portfolio with quantity > 0
```
- On app load, compile the list of active PSE stock symbols (holdings where `quantity > 0` and `market = PSE`).
- Only call the EODHD price endpoint for symbols in this active list.
- Do **not** fetch prices for: sold-out positions (quantity = 0), watchlist-only items (if any), or any symbol not currently held.

#### B. Cache fetched prices
- After fetching prices, store them in a server-side cache (Redis, in-memory store, or database field) with a **timestamp**.
- Cache TTL (time-to-live): **End of trading day** (PSE closes at 3:30 PM Philippine time, GMT+8). After market close, cached prices are valid until the next trading day open (9:30 AM PHT).
- On subsequent page loads within the same trading session, serve cached prices — do not re-call EODHD.
- Only invalidate cache and re-fetch when: cache is expired, user explicitly has no cached data (first load), or a new active symbol is added to the portfolio.

#### C. Remove the 20 API call per day limit
- Find and delete the code that enforces a hard cap of 20 API calls per day.
- The only rate limiting should be natural (cache-based) — fetch each symbol once per trading day, not artificially capped.

#### D. Fetch only price data
- When calling EODHD, request **only the fields needed**: `close` (or `adjusted_close`) price, `date`, and `symbol`.
- Do not request fundamentals, financials, dividend schedules, or other metadata from EODHD unless those are used elsewhere in the app.
- Use the EODHD **EOD (End of Day) endpoint** or **real-time/live endpoint** depending on subscription tier — confirm with the EODHD API docs which endpoint returns the latest available price.

**Sample optimized fetch logic (pseudocode):**
```
function getPSEPrices(activeSymbols):
  cachedData = cache.get('pse_prices')
  
  if cachedData exists AND cachedData.timestamp > lastMarketOpen:
    return cachedData.prices  // serve from cache
  
  // Only fetch active symbols
  symbolsToFetch = activeSymbols.filter(s => s.market == 'PSE' && s.quantity > 0)
  
  // Batch request if EODHD supports it, else loop
  prices = EODHD.fetchBatch(symbolsToFetch, fields=['close', 'date'])
  
  cache.set('pse_prices', { prices, timestamp: now() })
  return prices
```
